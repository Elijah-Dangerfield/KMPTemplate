Architecture notes: 
- We have a highly modularized app. 3 app types. App, Library, Feature. App depends on everything and is the entry point/glue. Features are what they sound like. They describe some functionality of the app that isn’t core to the app. Like how “casting” is a feature of a streaming app. But the idea of a “movie” isn’t a feature its critical to the app. Thats where libraries come in. They house core concepts. Features usually depend on multiple libraries. Libraries never depend on features. But features can depend on each other. Every library and feature is split into the main module and the sub module “impl”. The main module only exposes interfaces and domain models. The implementation obviously handles the implementation. Implementation modules can only depend on api modules. Generally only app depends on the implementation modules. And thats just because we use kotlin-inecjt-anvil to bind the impls. If a feature or library needs to access the DB it can depend on the storage module but it should also make its own storage sub module that it will use to define entities and DAOs. And bind them into the graph. the DB module will depend on the feature or libraries storage module to add the DAO and entity to the DB. 
- We dont have alot of use for expect and actual. If we have 2 different impls we will just bind 1 in android and another in iOS. If the iOS implementation can’t be written in kotlin then we implement it in iOS and pass it into the main view controller which just takes it and adds it to the DI graph.
